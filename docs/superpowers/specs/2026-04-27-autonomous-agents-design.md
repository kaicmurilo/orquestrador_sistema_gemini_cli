# Autonomous Agents System — Design Spec
Date: 2026-04-27

## Overview

Pipeline of autonomous Python agents coordinated by a central orchestrator. First agent researches market opportunities via Gemini CLI web search, then planning agents generate structured output files, then executor agents build a complete system from those files.

---

## 1. File Structure

```
agentesAutonomos/
├── orchestrator.py
├── tasks.json                    # global task state (runtime)
├── agents/
│   ├── research_agent.py
│   ├── plan_agent.py
│   ├── code_agent.py
│   └── test_agent.py
├── output/
│   ├── idea.md
│   ├── architecture.md
│   ├── stack.md
│   └── tasks.json                # generated by plan_agent
├── workspace/                    # generated project goes here
└── logs/                         # per-agent timestamped logs
```

---

## 2. Execution Flow

```
python orchestrator.py
  → research_agent.py
      Gemini web search: market trends, gaps, validation
      Output: output/idea.md
      Signals: task "research" = done

  → plan_agent.py
      Reads: output/idea.md
      Gemini generates: output/idea.md, output/architecture.md, output/stack.md, output/tasks.json
      output/tasks.json contains typed task list for executor agents
      Signals: task "planning" = done

  → orchestrator reads output/tasks.json
      For each task (respecting depends_on):
        dispatch code_agent.py  (type: "code")
          Reads: architecture.md + stack.md + specific task
          Gemini generates code → saves to workspace/
          Marks task done → signals orchestrator

        dispatch test_agent.py  (type: "test")
          Validates code_agent output
          On failure: creates new "fix" task in queue

  → all tasks done → pipeline complete
```

## 3. tasks.json Schema

```json
{
  "tasks": [
    {
      "id": "task_001",
      "type": "code",
      "status": "pending",
      "description": "Create authentication API",
      "depends_on": [],
      "output_file": "workspace/auth/api.py",
      "retries": 0,
      "error": null
    }
  ]
}
```

**Task types:** `research`, `planning`, `code`, `test`, `fix`

**Task statuses:** `pending`, `in_progress`, `done`, `failed`, `blocked`

---

## 4. Gemini CLI Integration

Each agent calls Gemini via subprocess with `--yolo` flag enabled globally. `--yolo` disables confirmation prompts, enables web search, and allows unrestricted file system modifications — required for agents to read/write workspace files autonomously.

```python
import subprocess

def call_gemini(prompt: str, allow_modifications: bool = True) -> str:
    cmd = ["gemini", "-p", prompt]
    if allow_modifications:
        cmd.append("--yolo")
    result = subprocess.run(cmd, capture_output=True, text=True)
    return result.stdout.strip()
```

All agents use `allow_modifications=True` by default. Pass `False` only for read-only queries where file system access is not needed.

Agents signal completion by updating `tasks.json` directly. Orchestrator runs a polling loop detecting status changes and dispatching next ready tasks.

---

## 5. Error Handling & Retry

**Agent failure** (timeout, empty output, subprocess error):
- Agent marks task `status: "failed"` with error message
- Orchestrator retries up to `MAX_RETRIES = 3`
- After 3 failures: marks task `blocked`, logs, halts pipeline

**Invalid output** (malformed content from Gemini):
- `research_agent` / `plan_agent`: validate file exists and has minimum content
- `code_agent`: `test_agent` detects and creates a `"fix"` task automatically

**Fix cycle limit:** fix tasks have `max_fix_cycles: 2` — exhausted = `blocked`

**Logging:** each agent writes to `logs/<agent_name>_<timestamp>.log`

---

## 6. Agent Responsibilities

| Agent | Input | Output |
|---|---|---|
| `research_agent` | none (web search) | `output/idea.md` |
| `plan_agent` | `output/idea.md` | `output/architecture.md`, `output/stack.md`, `output/tasks.json` |
| `code_agent` | architecture.md + stack.md + task | `workspace/<file>` |
| `test_agent` | workspace file + task | pass/fail + optional "fix" task |

---

## 7. Phased Delivery

**Phase 1 (this spec):** research + planning pipeline — orchestrator + research_agent + plan_agent only.

**Phase 2:** executor agents — code_agent + test_agent + full task queue loop.

**Phase 3:** additional specialized agents (e.g., `docs_agent`, `deploy_agent`).
